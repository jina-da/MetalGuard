## 1. 개요

MetalGuard Arduino Bridge Manager는 금속 탐지 판정 시스템의 **중간 브리지 역할**을 합니다.

- 운용 서버(Linux)로부터 판정 결과(`VERDICT`)를 수신하여 아두이노에 전달합니다.
- 아두이노로부터 동작 완료 신호(`DONE`)를 수신하여 운용 서버에 전달합니다.
- 연결이 끊어지면 자동으로 재접속을 시도합니다.

**개발 환경**

| 항목 | 내용 |
|------|------|
| 언어 | C++ (MFC) |
| IDE | Visual Studio 2019 이상 |
| 타겟 OS | Windows 10 이상 |
| 아두이노 통신 | Serial (9600 baud) |
| 서버 통신 | TCP/IP (Big-Endian) |

**폴더 구성**
이 저장소는 세 개의 폴더로 구성되어 있습니다.

| 폴더 | 설명 |
|------|------|
| `MetalGuard/` | 아두이노에 업로드하는 펌웨어 코드 (`MetalGuard.ino`) |
| `MetalGuardTerminal/` | 브리지 프로그램 초기 버전 (Windows 콘솔 기반, 현재 미사용) |
| `MetalGuardUI/` | 브리지 프로그램 현재 버전 (MFC GUI 기반, 현재 운용 중) |

### 터미널 버전에서 UI 버전으로 전환한 이유

초기에는 개발 속도를 높이기 위해 콘솔 환경(`MetalGuardTerminal`)에서 브리지 프로그램을 구현하였습니다. 콘솔 버전은 핵심 통신 로직 검증에는 충분했으나, 실제 운용 환경에서 다음과 같은 한계가 있었습니다.

**1. 상태 파악의 어려움**

콘솔에는 서버 연결 상태, 아두이노 연결 상태, 마지막 통신 시각 등이 로그 스크롤 속에 묻혀 현재 상태를 한눈에 파악하기 어려웠습니다. 운용 중 이상 발생 여부를 즉시 인지하려면 별도의 상태 표시 영역이 필요했습니다.

**2. 서버와 아두이노 로그의 혼재**

콘솔에서는 서버 수신 로그와 아두이노 수신 로그가 시간 순서로 뒤섞여 출력됩니다. 문제 발생 시 어느 쪽에서 이상이 생긴 것인지 추적하기 위해 로그를 거슬러 올라가야 하는 불편함이 있었습니다. UI 버전에서는 서버 로그와 아두이노 로그를 별도 창에 분리하여 각 채널의 상태를 독립적으로 확인할 수 있습니다.

**3. 설정 변경의 번거로움**

콘솔 버전에서 서버 IP, 포트, COM 포트를 변경하려면 소스 코드를 수정하고 재빌드해야 했습니다. 테스트 환경과 운용 환경의 IP가 다른 상황에서 이는 불필요한 작업을 유발합니다. UI 버전에서는 실행 중에 설정값을 직접 입력하고 즉시 적용할 수 있습니다.

**4. 수동 테스트 방식의 개선**

터미널 버전에도 `test.cpp`를 통해 서버 없이 아두이노에 직접 신호를 전송하는 테스트 기능이 존재했습니다. 그러나 이 방식은 콘솔에서 명령어를 직접 타이핑(`scanf`)해야 하고, 메인 브리지 프로그램과 별개로 실행해야 하는 구조였습니다. 즉 브리지를 운용하면서 동시에 수동 테스트를 수행하려면 두 개의 프로세스를 병행 관리해야 했습니다. UI 버전에서는 PASS / FAIL / UNCERTAIN / TIMEOUT 버튼이 브리지 프로그램 내에 통합되어, 운용 중에도 별도 프로세스 없이 즉시 신호를 전송할 수 있습니다.

**5. 사용자를 위한 운용 편의성 확보**

실제 운용 환경에서는 현장 담당자가 프로그램을 직접 실행하고 상태를 모니터링해야 하는 경우가 생길 수 있습니다. 콘솔 환경은 로그의 흐름만으로 현재 상태를 파악해야 하므로, 시스템에 익숙하지 않은 사용자에게는 어려움이 따릅니다. UI 버전은 연결 상태, 로그, 설정값을 시각적으로 구분된 영역에 배치하여 터미널에 익숙하지 않은 사용자도 별도의 교육 없이 프로그램을 실행하고 상태를 확인할 수 있습니다.

이러한 이유로 콘솔 버전을 기반으로 통신 엔진(`Local.cpp`)은 그대로 유지하되, MFC 다이얼로그 기반의 UI를 추가하여 `MetalGuardUI`로 발전시켰습니다. 터미널 버전은 참조 및 히스토리 보존 목적으로 저장소에 함께 유지합니다.

## 3. 통신 프로토콜

### 3-1. 패킷 구조 (서버 ↔ PC)

모든 패킷은 다음 헤더로 시작합니다. 서버(Linux)는 Big-Endian, PC(Windows)는 Little-Endian이므로 바이트 순서 변환이 자동 처리됩니다.

```
┌──────────────┬──────────────┬──────────────────────────┐
│  signature   │    cmdId     │        bodySize          │
│  2 bytes     │  2 bytes     │        4 bytes           │
│  0x4D47      │  (CmdID)     │  (이후 본문 바이트 수)      │
└──────────────┴──────────────┴──────────────────────────┘
```

### 3-2. CmdID 목록

| CmdID | 이름 | 방향 | 설명 |
|-------|------|------|------|
| 201 | VERDICT_PASS | 서버 → PC → 아두이노 | 통과 판정 |
| 202 | VERDICT_FAIL | 서버 → PC → 아두이노 | 실패 판정 |
| 203 | VERDICT_UNCERTAIN | 서버 → PC → 아두이노 | 불확실 판정 |
| 204 | VERDICT_TIMEOUT | 서버 → PC → 아두이노 | 타임아웃 판정 |
| 205 | DONE_PASS | 아두이노 → PC → 서버 | 통과 동작 완료 |
| 206 | DONE_FAIL | 아두이노 → PC → 서버 | 실패 동작 완료 |
| 207 | DONE_UNCERTAIN | 아두이노 → PC → 서버 | 불확실 동작 완료 |
| 208 | DONE_TIMEOUT | 아두이노 → PC → 서버 | 타임아웃 동작 완료 |
| 501 | PING | PC → 서버 | 연결 확인 요청 |
| 502 | PONG | 서버 → PC | 연결 확인 응답 |

### 3-3. 아두이노 시리얼 신호

| 방향 | 신호 | 의미 |
|------|------|------|
| PC → 아두이노 | `P\n` | VERDICT_PASS 전달 |
| PC → 아두이노 | `F\n` | VERDICT_FAIL 전달 |
| PC → 아두이노 | `U\n` | VERDICT_UNCERTAIN 전달 |
| PC → 아두이노 | `T\n` | VERDICT_TIMEOUT 전달 |
| PC → 아두이노 | `H\n` | Heartbeat (5초 간격) |
| 아두이노 → PC | `P_done` | DONE_PASS 신호 |
| 아두이노 → PC | `F_done` | DONE_FAIL 신호 |
| 아두이노 → PC | `U_done` | DONE_UNCERTAIN 신호 |
| 아두이노 → PC | `T_done` | DONE_TIMEOUT 신호 |
| 아두이노 → PC | `PONG` | Heartbeat 응답 |

## 4. UI 사용 방법

### 4-1. 연결 설정

프로그램 상단의 **연결 설정** 영역에서 접속 정보를 입력합니다.

| 항목 | 설명 | 기본값 |
|------|------|--------|
| 서버 IP | 운용 서버의 IP 주소 | 10.10.10.109 |
| 포트 | 운용 서버의 TCP 포트 | 8000 |
| 아두이노 포트 | PC에 연결된 COM 포트 | 자동 탐색 |

> **아두이노 포트 드롭다운**을 클릭할 때마다 현재 연결된 COM 포트 목록이 자동으로 갱신됩니다.

### 4-2. 연결 / 정지

- **[연결] 버튼**: 입력된 설정으로 브리지를 시작합니다. 아두이노와 서버 모두 자동으로 연결을 시도하며, 실패해도 3초 간격으로 재시도합니다.
- **[정지] 버튼**: 브리지를 즉시 중단합니다.

### 4-3. 로그 화면

화면 중앙에 **서버 로그**와 **아두이노 로그** 두 개의 창이 나란히 표시됩니다.

| 로그 창 | 표시 내용 |
|---------|----------|
| 서버 로그 | 서버와의 송수신 내용 및 연결 상태 변경 |
| 아두이노 로그 | 아두이노와의 송수신 내용 및 연결 상태 변경 |

각 창 아래의 **[로그 지우기]** 버튼으로 해당 창의 로그를 초기화할 수 있습니다.

### 4-4. 시스템 상태

화면 하단의 **시스템 상태** 영역에서 현재 연결 상태를 1초 간격으로 확인할 수 있습니다.

| 항목 | 설명 |
|------|------|
| 운용서버 연결 상태 | 서버와의 TCP 연결 여부 |
| 아두이노 연결 상태 | 마지막 PONG 수신 후 15초 이내이면 연결됨으로 표시 |
| 아두이노 연결 확인 시간 | 마지막으로 PONG을 수신한 시각 |

### 4-5. 수동 조작 (Debug)

화면 최하단의 버튼으로 아두이노에 판정 신호를 직접 전송할 수 있습니다. 서버 없이 아두이노 동작을 테스트할 때 사용합니다.

| 버튼 | 전송 신호 |
|------|----------|
| [PASS] | `P\n` |
| [FAIL] | `F\n` |
| [UNCERTAIN] | `U\n` |
| [TIMEOUT] | `T\n` |

## 5. 로그 읽는 방법

모든 로그는 `[HH:MM:SS]` 형식의 타임스탬프와 함께 출력됩니다.

```
[10:04:44] [시작] MetalGuard 아두이노 브리지 시작
[10:04:44] [아두이노 연결] 연결 시도 (COM3)
[10:04:44] [아두이노 연결] 아두이노 응답 확인!     ← 실제 아두이노 통신 확인
[10:04:44] [서버 연결] 연결 성공! (10.10.10.109:8000)
[10:04:44] ◁ [서버 전송] CmdID 501 / PING
[10:04:44] ▶ [서버 수신] CmdID 502 / PONG
[10:04:50] ▶ [서버 수신] CmdID 201 / VERDICT_PASS   ← 서버로부터 판정 수신
[10:04:50] ◁ [아두이노 전송] CmdID 201 / VERDICT_PASS
[10:04:51] ▶ [아두이노 수신] CmdID 205 / DONE_PASS  ← 아두이노 동작 완료
[10:04:51] ◁ [서버 전송] CmdID 205 / DONE_PASS
```

**로그 기호 설명**

| 기호 | 의미 |
|------|------|
| `▶` | 수신 (데이터가 들어옴) |
| `◁` | 송신 (데이터를 보냄) |

## 6. 설정 변경 방법

코드에서 변경이 필요한 값들은 `Local.cpp` 상단에 모여 있습니다.

### 신호 문자열 변경

아두이노와 주고받는 시리얼 신호를 바꾸려면 `SIGNAL_TABLE`만 수정합니다.

```cpp
// Local.cpp
const SignalMap SIGNAL_TABLE[] = {
    { VERDICT_PASS,      "P\n",   nullptr  },  // 서버→아두이노: "P\n" 전송
    { DONE_PASS,         nullptr, "P_done" },  // 아두이노→서버: "P_done" 수신
    // ...
};
```

### 타임아웃 / 재접속 간격 변경

```cpp
// Local.cpp
const int PONG_TIMEOUT_SEC    = 8;  // 아두이노 무응답 판정 시간 (초)
const int RECONNECT_DELAY_SEC = 3;  // 재접속 시도 간격 (초)
```

### 서버 기본 IP / 포트 변경

UI에서 직접 입력할 수 있지만, 프로그램 시작 시 기본값을 바꾸려면 `MetalGuardUIDlg.cpp`의 `OnInitDialog`를 수정합니다.

```cpp
// MetalGuardUIDlg.cpp - OnInitDialog()
m_editServerIp.SetWindowTextW(_T("10.10.10.109"));  // 기본 IP
m_editPort.SetWindowTextW(_T("8000"));               // 기본 포트
```

## 7. 소스 파일 구조

```
MetalGuardUI/
├── Local.cpp              # 브리지 핵심 로직 (통신, 스레드, 신호 처리)
├── MetalGuardUI.cpp       # MFC 앱 진입점 (CWinApp)
├── MetalGuardUI.h
├── MetalGuardUIDlg.cpp    # 메인 다이얼로그 UI 로직
├── MetalGuardUIDlg.h
├── MetalGuardUI.rc        # UI 리소스 (다이얼로그 레이아웃)
├── resource.h             # 컨트롤 ID 정의
├── Serial.cpp             # 시리얼 통신 보조 클래스 (미사용)
├── SerialClass.h
├── pch.h / pch.cpp        # 미리 컴파일된 헤더
├── framework.h            # MFC 헤더 모음
└── targetver.h            # Windows SDK 버전 설정
```

**핵심 파일 역할 요약**

`Local.cpp`는 UI와 완전히 분리된 통신 엔진으로, 세 개의 스레드로 구성됩니다.

| 스레드 | 역할 |
|--------|------|
| `BridgeMain` | 아두이노 포트 열기 → 서버 연결 → 수신 스레드 시작 |
| `serverRecvLoop` | 서버로부터 패킷 수신 → 아두이노에 신호 전달 |
| `arduinoRecvLoop` | 아두이노로부터 시리얼 수신 → 서버에 패킷 전달 |
| `heartbeatThread` | 5초 간격으로 `H` 전송 → PONG 무응답 시 아두이노 재접속 |

## 8. 주요 동작 흐름

### 정상 동작

```
[연결 버튼 클릭]
       │
       ├───────────────────────────────────────────────────────────────┐
       ▼                                                               ▼
아두이노 포트 열기 시도 ──실패──► 3초 후 재시도                     서버 TCP 연결 시도 ──실패(5초 타임아웃)──► 3초 후 재시도
       │ 성공                                                           │ 성공
       ▼                                                               ▼
heartbeatThread 시작 (5초 간격 H 전송)                           PING 전송 → PONG 수신
       │                                                               │
       ▼                                                               │
서버 TCP 연결 시도 ──실패(5초 타임아웃)──► 3초 후 재시도                     │
       │ 성공                                                           │
       ▼                                                               │
PING 전송 → PONG 수신                                                    │
       │───────────────────────────────────────────────────────────────┘
       ▼
    대기 중
       │
       ├─ 서버에서 VERDICT 수신 ─► 아두이노에 시리얼 신호 전송
       │
       └─ 아두이노에서 DONE 수신 ─► 서버에 패킷 전송
```

> 서버 연결이 끊겨 재연결을 시도하는 동안에도 heartbeatThread는 계속 동작하여 아두이노와의 연결을 독립적으로 유지합니다. (REQ-A03)

### Heartbeat 흐름

```
PC ──── H\n ────► 아두이노
PC ◄─── PONG ──── 아두이노   → g_lastPongTime 갱신

8초 이상 PONG 없음
       │
       ▼
아두이노 핸들 닫기 → 3초 후 포트 재열기 시도 → 연결 시도 로그 출력
PONG 수신 시 → "아두이노 응답 확인!" 출력
```

## 9. 자동 재연결 동작

| 상황 | 동작 |
|------|------|
| 서버 연결 실패 | 논블로킹 `select()`로 5초 타임아웃 → 실패 처리 → 3초 후 재시도 (무한 반복) |
| 서버 연결 끊김 | 즉시 감지 → 3초 후 재연결 시도 (아두이노 연결은 독립 유지) |
| 아두이노 포트 열기 실패 | 3초 후 재시도 (무한 반복) |
| 아두이노 PONG 무응답 (8초) | 포트 닫기 → 3초 간격 재시도 → PONG 수신 시 연결 확인 출력 |

서버와 아두이노의 재연결은 **독립적으로 동작**합니다. 서버가 끊겨도 아두이노 Heartbeat는 계속 유지됩니다.

> **connect() 타임아웃 방식**: 소켓을 논블로킹 모드로 전환 후 `connect()`를 호출하고, `select()`로 최대 5초 대기합니다. OS 기본 타임아웃(~20초)을 사용하지 않아 빠른 재시도가 가능합니다. (REQ-A03)

## 10. 자주 묻는 질문

**Q. 아두이노 포트가 드롭다운에 보이지 않습니다.**
드롭다운을 한 번 클릭하면 COM 포트 목록이 자동으로 갱신됩니다. 그래도 보이지 않으면 장치 관리자에서 아두이노가 정상적으로 인식되었는지 확인하세요.

**Q. 아두이노 연결 상태가 "연결 끊김"으로 표시됩니다.**
아두이노가 Heartbeat(`H`)에 `PONG`으로 응답해야 합니다. 아두이노 스케치에 PONG 응답 코드가 포함되어 있는지 확인하세요. 마지막 PONG 수신 후 15초가 지나면 끊김으로 표시됩니다.

**Q. 서버 로그에 재연결 시도가 보이지 않습니다.**
잘못된 IP로 `connect()`를 시도하면 최대 5초간 블로킹됩니다. 5초 타임아웃 후 "연결 실패" 로그가 출력되고 3초 후 재시도합니다.

**Q. 수동 조작 버튼이 동작하지 않습니다.**
아두이노가 연결된 상태(포트가 열린 상태)에서만 동작합니다. 아두이노 연결 상태를 확인하세요.

**Q. 신호 문자열을 변경하려면 어떻게 하나요?**
`Local.cpp`의 `SIGNAL_TABLE`만 수정하면 됩니다. UI나 다른 로직은 변경할 필요가 없습니다.
